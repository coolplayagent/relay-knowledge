# Storage Engine and Mutation Log

## 1. Design Conclusion

The storage layer is a graph state machine, not a data-access helper. SQLite-first is the default product route because it provides local zero-dependency operation, transactions, WAL, FTS5, recursive CTEs, and low test cost. Business layers depend only on storage traits, not SQLite details.

## 2. Storage Boundary

```text
Application Services
        |
        v
Storage Facade: GraphStore, MutationLogStore, IndexStore, CodeGraphStore
        |
        +--> SQLite Adapter: WAL, transactions, FTS5, CTE
        +--> Future Adapters: SurrealDB, Neo4j, NebulaGraph, Memgraph
```

`domain` does not depend on SQL or connection pools; `retrieval` does not bypass the storage facade; `interfaces` do not duplicate storage logic.

In service deployments, the primary runtime database is the control-plane state machine for tasks, leases, audit, operator state, topology catalog, repository membership, and diagnostics. Code-fact shards, derived indexes, and external backends are data-plane implementations. The control plane and data plane may be physically separated, but the call boundary remains application services plus the storage facade; interface layers and workers must not directly access concrete backends.

## 3. Write Transaction

Graph writes follow a fixed flow:

```text
validate domain model
  -> begin transaction
  -> upsert evidence/entities/relations/claims/events/code facts
  -> append graph_mutations
  -> bump graph_version
  -> mark affected index cursors stale
  -> commit
  -> publish refresh work
```

Graph facts, mutation log entries, and graph version bumps are in the same transaction. Index refresh work is published only after commit.

## 4. Mutation Log

The mutation log is the spine of index recovery and audit. Each mutation includes at least graph version, affected scopes, affected entities, affected evidence, source hashes, fact kinds, index families, actor/runtime identity, and trace id.

Indexers consume mutation logs and scoped cursors; they do not scan the entire database to infer refresh work.

## 5. SQLite Runtime Model

SQLite operations may use a synchronous driver, but they are isolated behind blocking workers, dedicated connections, or pool boundaries and never occupy async executors. Writes use batch transactions; reads use prepared statements and compound indexes; FTS candidate windows apply scope/path/language filters before scoring.

## 6. SQLite Storage Topologies

The default topology is `single_sqlite`: graph facts, mutation logs, worker state, audit records, file indexes, and code-repository facts share one runtime database. The optional `partitioned_sqlite` topology is the v1 data-plane expansion implementation: it keeps the same storage trait contract, leaves global control state in the main database, and routes each repository's files, symbols, references, chunks, checkpoints, and scoped code queries to an independent SQLite shard. Shard files are generated by `paths` under the runtime data directory at `stores/repositories/<safe-id-hash>/code.sqlite`; they are not written to source repositories, current working directories, or release extraction directories.

In the partitioned topology, durable tasks, leases, dead letters, audit, graph facts, and repository-set membership remain owned by the control database so the at-most-one active writer task per repository invariant is not bypassed. Cross-repository overlay refresh must explicitly report unavailable until cross-shard import/export aggregation exists; it must not fake freshness or copy base code facts into the control database.

A runtime database with active partitioned shard catalog rows is no longer a valid `single_sqlite` database. Startup with `single_sqlite` must fail fast instead of returning repository status from the control database while code facts remain in shard files.

The shard catalog stores only relocatable routing metadata. Readers verify active catalog rows but recompute shard file paths from `RuntimePaths` and `repository_id`, so backup restore or runtime-home moves do not depend on obsolete absolute paths.

Scope migration into a shard must preserve the code facts, checkpoint progress,
and derived software projection rows for that scope, or keep the scope routed to
the control database until those derived rows are refreshed.

## 7. Backend Evolution

Future graph database, vector database, replicated SQLite, or remote storage adapters implement the same contract tests. New backends do not change the domain fact model, mutation log semantics, freshness contract, degraded reasons, or stable error kinds; they change only persistence, sharding, indexing, and query primitives.

## 8. Acceptance Criteria

- A storage write either commits facts, mutation, and version completely or rolls back completely.
- Index failure does not roll back graph writes, but it creates stale/degraded diagnostics.
- SQLite-specific optimizations do not leak into domain, API, or interface types.
- Partitioned topologies do not bypass durable task leases, bounded retry/backoff, checkpoint replay, or per-repository writer constraints.
- Control-plane/data-plane split deployments do not let workers, Web, MCP, or CLI bypass the storage facade to read or write shards or external backends directly.
